Ordinary inline code work stays on the current user model; do not launch Spark merely because a task is small. When full `task-analyze-skill` has explicit benchmark evidence and admits delegation, its locked plan is the authority for model and effort. In that admitted route, use `GPT-5.3-Codex-Spark` (`gpt-5.3-codex-spark`) first only for obvious bounded, low-risk, easy, low-ambiguity text-only tiny implementation, edit, repair/refactor, command, or authored-probe work in an active registry-owned code domain. This is a cold-start exception, not a permanent code-task assignment.

## Capability Contract

- Inputs: text only; never use Spark for image reading.
- Context: 128,000 tokens in the cached local catalog.
- Supported efforts: low, medium, high, xhigh.
- API-only execution: unsupported in the cached local catalog.
- A model name in a plan or availability check is not proof. Require a runtime receipt before claiming Spark ran.

## Use When

- The node is an active registry-owned code domain and loads `code-skill`; current examples are Python, plain C#, and Unity C#.
- The work is tiny and fits Spark context, does not require image input, repository breadth, integration evidence, or unresolved cross-system judgment.
- A positively admitted plan already defines sources, output, stop condition, post-result Ending Real Verify, and fallback.

## Visible Fallback

- Spark-low unavailable, exhausted, rejected, or unsupported on the surface: use the learner-selected pair from the full normal fallback ladder; never raise Spark effort.
- Image, context, repository breadth, integration, or verification complexity caused the mismatch: use the planned Terra fallback.
- Record `from`, `to`, reason, and effort. Never silently substitute the entry/active model.
- If selection metadata is unavailable, report the planned model and limitation; do not claim execution.

Spark never bypasses Qin's code style, `code-skill`, project rules, or post-result Ending Real Verify. Present the completed result first with no foreground Mini/Fast Verify; a different Ending Task verifier remains required for optimization.

The tiny candidate ladder is exactly Spark-low plus the full normal fallback. Every non-tiny code route—easy or complex—uses the exact full normal ladder with no Spark. `code-skill` remains mandatory throughout.
